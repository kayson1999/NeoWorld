# NeoWorld — 策略生存类游戏 项目计划书

## 一、项目概述

**NeoWorld** 是一款以 AI 驱动叙事为核心的策略生存类游戏。玩家通过对话决策和情感输入，塑造角色的性格、行为与命运。游戏构建在一套自定义的世界模型之上，融合经济系统、生存法则与社交交互，支持多人协同游玩和插件化扩展。

### 核心理念

- **叙事双驱动**：被动故事线（游戏内预设剧情分支） + 主动故事线（玩家真实情感输入影响角色命运）
- **世界模型自洽**：经济系统、生存规则、社会关系构成闭环
- **多人协同**：支持多玩家共存于同一世界，实时交互与社交
- **可扩展架构**：插件系统支持第三方内容与玩法扩展

## 三、核心系统设计

### 3.1 世界模型（World Model）

世界模型是整个游戏的基础，定义游戏发生的环境与规则。

| 模块 | 说明 |
|------|------|
| **环境定义** | 世界背景设定（赛博朋克城市 / 末日废土 / 架空社会 等），地图结构、区域划分 |
| **世界规则引擎** | 类似"一个老板 N 个员工"的社会结构，定义角色间的层级、竞争、合作关系 |
| **时间系统** | 日夜循环、季节变化、事件触发时间线 |
| **状态机** | 世界状态的有限状态机，驱动全局事件和阶段转换 |

### 3.2 故事引擎（Story Engine）

双线叙事是本游戏最核心的差异化特性。

**被动故事线（游戏内决策）：**
- 预设剧情树，节点为决策点
- 玩家选择不同选项 → 分支走向不同结局
- 决策因子：角色当前状态、历史选择、与 NPC 关系

**主动故事线（情感机器人）：**
- 玩家在对话框中自由输入真实感受/遭遇
- LLM 分析情感 → 映射为游戏内事件
- 长期记忆存储，持续影响角色命运弧线

```
用户输入 → 情感分析 → 情感向量 → 决策权重调整 → 世界事件触发
                                ↓
                         角色性格/命运变化
```

### 3.3 经济与生存模型

| 层级 | 内容 |
|------|------|
| **资源层** | 定义稀缺资源（货币、食物、材料、信息） |
| **生产层** | 资源获取方式（工作/采集/交易/掠夺） |
| **消耗层** | 生存开销（食物、住所、安全） |
| **淘汰机制** | 资源低于阈值 → 触发生存危机事件 → 可能导致角色"死亡"或降级 |
| **经济平衡** | 参考经济学论文（供需模型、博弈论），防止通胀/通缩 |

### 3.4 AI 系统

| 模块 | 技术方案 |
|------|----------|
| **NPC 行为** | 行为树 + 有限状态机，支持复杂行为链 |
| **对话系统** | LLM API（GPT/Claude/本地模型），上下文记忆管理 |
| **情感引擎** | 情感维度模型（PAD模型：愉悦-唤醒-支配），实时情绪计算 |
| **决策系统** | 加权决策树，综合角色性格 + 情感状态 + 外部环境 |

### 3.5 多人协同架构

```
┌─────────┐     ┌─────────┐     ┌─────────┐
│ Client A │     │ Client B │     │ Client C │
└────┬─────┘     └────┬─────┘     └────┬─────┘
     │                │                │
     └───────────┬────┘────────────────┘
                 ▼
        ┌─────────────────┐
        │   Game Server    │  ← 权威状态管理
        │  (Room Manager)  │
        ├─────────────────┤
        │  State Sync      │  ← 状态同步引擎
        ├─────────────────┤
        │  World Instance  │  ← 世界模型实例
        ├─────────────────┤
        │  AI Service      │  ← AI/LLM 服务
        ├─────────────────┤
        │  DB / Cache      │  ← 持久化 + 缓存
        └─────────────────┘
```

**同步策略：**
- 关键状态（经济、生存值）：服务器权威 + 状态同步
- 非关键状态（UI 动画）：客户端本地处理
- 对话/社交：消息队列 + 异步同步

## 四、难点分析与解决思路

### 难点 1：世界模型与场景构建

| 挑战 | 解决思路 |
|------|----------|
| 世界规则的自洽性 | 使用形式化规则引擎（如有限状态机 + 约束求解），确保规则无矛盾 |
| 场景丰富度 | 数据驱动场景生成，场景模板 + 参数化配置，减少硬编码 |
| 动态世界 | ECS（Entity-Component-System）架构，世界状态可组合、可序列化 |

### 难点 2：经济/生存模型平衡

| 挑战 | 解决思路 |
|------|----------|
| 经济平衡 | 参考 Cobb-Douglas 生产函数 + 供需曲线，模拟宏观经济 |
| 通胀/通缩 | 引入动态调节器（类央行机制），自动调整资源产出/消耗率 |
| 多人公平性 | 区分 PvE/PvP 经济区，防止滚雪球效应 |

### 难点 3：用户体验增强

| 挑战 | 解决思路 |
|------|----------|
| 对话质量 | LLM + RAG（检索增强生成），结合世界观知识库 |
| 被动→主动 | 推送系统（世界事件通知）、NPC 主动互动、定时情感问候 |
| 画面表现 | 分阶段：先文字 RPG → 2D 场景 → 逐步增强视觉效果 |
| 社交体验 | 公会系统、交易市场、竞技场、排行榜 |

## 七、协同开发规范

### Git 工作流
- `main` — 稳定版本
- `develop` — 开发集成
- `feature/*` — 功能分支
- `hotfix/*` — 紧急修复

### 代码规范
- 模块化设计，每个子系统独立可测试
- 接口先行，实现后置
- 数据驱动优于硬编码
- 所有配置抽离到 `data/configs/`

### 沟通与文档
- 设计文档放 `docs/`
- API 接口文档自动生成
- 变更日志维护
